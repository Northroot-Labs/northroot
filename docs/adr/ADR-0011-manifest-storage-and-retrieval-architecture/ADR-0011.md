# ADR-0011: Manifest Storage and Retrieval Architecture

Date: 2025-11-12  
Status: Proposed  
Domain: Storage

## Context

During implementation of ADR-0010-P04 (Real End-to-End Example), we identified several architectural gaps in manifest storage and retrieval:

1. **Chunk Set Retrieval**: The example computes overlap by re-reading previous CSV files instead of retrieving chunk sets from stored receipts/manifests. This is a workaround that doesn't scale to production.

2. **Manifest Serialization**: `ByteStreamManifest` doesn't implement `Serialize`, forcing manual JSON construction. This breaks canonicalization consistency (receipts use CBOR, manifests use JSON).

3. **Storage API Gaps**: Missing methods to retrieve chunk sets from stored receipts for overlap computation. The storage layer stores manifests but doesn't provide convenient access to chunk sets.

4. **Canonicalization Inconsistency**: Receipts use CBOR canonicalization (RFC 8949) per ADR-0002, but manifests are serialized as JSON, creating inconsistency in the storage layer.

These issues prevent proper production use where:
- Chunk sets must be retrieved from storage (not re-computed from source data)
- Manifests must be canonicalized consistently (CBOR like receipts)
- Storage must provide efficient chunk set access for overlap computation

## Decision Drivers

- **Consistency**: Manifests should use the same canonicalization as receipts (CBOR)
- **Efficiency**: Chunk sets should be retrievable from storage without re-processing source data
- **Completeness**: Storage API should provide all necessary methods for overlap computation
- **Production Readiness**: Architecture must support real-world usage patterns
- **Maintainability**: Avoid workarounds and manual serialization

## Decision

### 1. Add CBOR Serialization to ByteStreamManifest

**Problem**: `ByteStreamManifest` doesn't implement `Serialize`, forcing manual JSON construction.

**Solution**: Add `serde` derives and implement CBOR serialization consistent with receipts.

**Implementation**:
```rust
// crates/northroot-engine/src/cas.rs

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub struct ByteStreamManifest {
    pub manifest_root: String,
    pub manifest_len: u64,
    pub chunks: Vec<Chunk>,
    #[serde(flatten)]
    pub chunk_scheme: ChunkScheme,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub struct Chunk {
    pub id: String,
    pub offset: u64,
    pub len: u64,
    #[serde(with = "hex_serde")]
    pub hash: [u8; 32],
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
#[serde(tag = "type", rename_all = "snake_case")]
pub enum ChunkScheme {
    CDC { avg_size: u64 },
    Fixed { size: u64 },
}
```

**Canonicalization**: Use CBOR canonicalization (RFC 8949) via `northroot-receipts` canonicalization utilities.

**Storage**: Store manifests as CBOR-encoded bytes, not JSON.

### 2. Add Tiered Chunk Set Storage Architecture

**Problem**: Storage doesn't provide methods to retrieve chunk sets from receipts for overlap computation. Full chunk sets can be large and slow to retrieve.

**Solution**: Implement tiered chunk set storage with lightweight sketches in manifests and full sets in CAS (Content-Addressed Storage).

**Design Principles**:
- **Manifests = pointers + sketches**: Fast decisions, lightweight sync
- **External sets = truth**: Accurate overlap, large, GC'able
- **Tiered approach**: Inline sketch (always), inline set (if tiny), external set (default)

**Why This Design**:
- **Latency**: Decide "reuse vs recompute" from manifest sketch alone. Fetch full sets only on close calls.
- **Size & churn**: Manifests stay tiny (good for diffs, reviews, ADR links). Chunk sets versioned independently.
- **Cacheability**: Compact sketches are hot; full sets live in CAS with local LRU.
- **GC & retention**: Easier to expire large sets by policy without rewriting manifests.
- **Security**: Fewer secrets in manifests; large sets can live in restricted store.

**Tiered Storage Strategy**:

1. **Inline sketch (always)** — ~hundreds of bytes:
   - MinHash (k=128) for Jaccard similarity estimation
   - Optional Bloom filter for membership pre-filter
   - Optional Roaring bitmap summary if universe is stable

2. **Inline set (only if tiny)** — e.g., ≤ 4–8 KB or ≤ 256 IDs:
   - Direct array of chunk IDs in manifest
   - Avoids CAS lookup for small sets

3. **External set (default)** — Store as CAS object:
   - Path: `/.nr/cas/aa/bb/<sha256>.cbor`
   - Content-addressed by SHA-256 of chunk set
   - Optionally sharded for very large sets
   - LRU cache (1–5 GB typical)

**Decision Rule (when to inline full set)**:

For overlap check cost:
- Let `O_inline` = time to compute J from inline set
- Let `O_fetch` = RTT + deserialize of external set
- **Inline if** `O_fetch > τ` and `size ≤ threshold`; else reference

**Typical defaults**: `τ ≈ 2–5 ms`, `size ≤ 8 KB` or `≤ 256 IDs`

**Manifest Schema Update**:
```rust
// crates/northroot-engine/src/cas.rs

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub struct ByteStreamManifest {
    pub manifest_root: String,
    pub manifest_len: u64,
    pub chunks: Vec<Chunk>,
    pub chunk_scheme: ChunkScheme,
    
    // New: Overlap index for fast reuse decisions
    pub overlap_index: OverlapIndex,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub struct OverlapIndex {
    /// Algorithm used for sketch (minhash, bloom, roaring)
    pub algo: String,
    
    /// Compact sketch for fast Jaccard estimation
    pub sketch: SketchData,
    
    /// Inline set (only if tiny: ≤ 256 IDs or ≤ 8 KB)
    #[serde(skip_serializing_if = "Option::is_none")]
    pub inline_set: Option<Vec<String>>,
    
    /// External set reference (default for larger sets)
    #[serde(skip_serializing_if = "Option::is_none")]
    pub external: Option<ExternalSetRef>,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub struct SketchData {
    /// MinHash k parameter (typically 128)
    pub k: u32,
    /// MinHash hash seeds
    pub hashes: Vec<String>,
    /// Optional Bloom filter parameters
    #[serde(skip_serializing_if = "Option::is_none")]
    pub bloom: Option<BloomParams>,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq)]
pub struct ExternalSetRef {
    /// Content-addressed ID (SHA-256 hash of chunk set)
    pub cid: String,
    /// Size hint in bytes
    pub bytes: u64,
    /// Schema/encoding (cbor-v1, parquet-v3, roaring-v1)
    pub schema: String,
}
```

**Storage API**:
```rust
// crates/northroot-storage/src/traits.rs

pub trait ReceiptStore: Send + Sync {
    // ... existing methods ...
    
    /// Store chunk set in CAS (Content-Addressed Storage).
    ///
    /// Stores full chunk set as CBOR-encoded content-addressed object.
    /// Returns CID (content ID) for reference in manifest.
    ///
    /// # Arguments
    ///
    /// * `chunk_set` - Set of chunk IDs
    ///
    /// # Returns
    ///
    /// CID (SHA-256 hash) of stored chunk set
    ///
    /// # Errors
    ///
    /// Returns error if storage operation fails
    fn store_chunk_set_cas(
        &self,
        chunk_set: &HashSet<String>,
    ) -> Result<String, StorageError>;  // Returns CID
    
    /// Retrieve chunk set from CAS by CID.
    ///
    /// # Arguments
    ///
    /// * `cid` - Content-addressed ID (SHA-256 hash)
    ///
    /// # Returns
    ///
    /// Some(HashSet<String>) if found, None if not found
    ///
    /// # Errors
    ///
    /// Returns error if storage operation fails
    fn get_chunk_set_cas(
        &self,
        cid: &str,
    ) -> Result<Option<HashSet<String>>, StorageError>;
}
```

**Retrieval Path**:

1. **Fast path**: Compute J from sketch vs candidate sketch
   - If clearly above/below threshold → decide immediately
   - No CAS fetch needed

2. **Close call**: Fetch external set(s) from CAS
   - Check local LRU cache first
   - Fall back to remote CAS if needed
   - Compute exact Jaccard similarity

3. **Record**: Store J, decision, and costs in receipts

**Data Structures (Rust-friendly)**:

- **Sketch**: MinHash (k=128) for Jaccard; optional b-bit minhash to shrink
- **Bloom**: Membership pre-filter (low false-positive rate)
- **Exact set**: Roaring bitmap (fast, compact) or sorted varints
- **Storage**: CBOR for portability; optionally Parquet when sets are huge

**Defaults**:
- MinHash k=128, 16-byte seeds → ~2 KB sketch
- Inline full set if `|set| ≤ 256` or `≤ 8 KB`
- CAS path: `/.nr/cas/aa/bb/<sha256>.cbor` with LRU (1–5 GB)
- Precompute sketches during ingest; compute exact sets lazily

**Ops & GC**:
- Manifests are immutable once published (small)
- CAS objects get reference counts from manifests
- Unreferenced → GC after TTL
- Version external sets independently (same CID if identical set)

**Security & Provenance**:
- Put SHA-256 of external set in manifest
- Optional signature for provenance
- Proofs (later): receipts bind `{manifest_id, cid, J, costs}`

### 4. Update Example to Use Storage for Chunk Sets

**Problem**: Example re-reads CSV files to compute overlap instead of using stored chunk sets.

**Solution**: Update example to:
1. Store chunk sets when generating receipts
2. Retrieve chunk sets from storage for overlap computation
3. Remove workaround of re-reading CSV files

**Implementation**:
```rust
// examples/finops_cost_attribution/main.rs

// After generating receipt:
let chunk_set: HashSet<String> = current_chunks; // from tuples_to_chunk_ids
store.store_chunk_set(&rid, &chunk_set)?;

// When computing overlap:
let (overlap_j, prev_chunks) = if let Some(prev) = &prev_receipt {
    let prev_chunks = store.get_chunk_set(&prev.rid)?
        .unwrap_or_default();
    let j = jaccard_similarity(&current_chunks, &prev_chunks);
    (j, prev_chunks)
} else {
    (0.0, HashSet::new())
};
```

## Implementation Plan

Implementation is decomposed into explicit phases ranked by priority. See phase files in `phases/` directory for detailed specifications.

### Phase Priority Summary

1. **P01 (HIGHEST)**: CBOR Serialization Foundation
   - Add CBOR serialization to `ByteStreamManifest`
   - Foundation for all other phases
   - **Blocks**: All other phases

2. **P02 (HIGH)**: Overlap Index Sketch Generation
   - Add MinHash sketch generation to manifests
   - Enable fast-path reuse decisions
   - **Depends**: P01

3. **P03 (HIGH)**: CAS Storage for Chunk Sets
   - Implement content-addressed storage for chunk sets
   - Add CAS storage methods to `ReceiptStore`
   - **Depends**: P01

4. **P04 (MEDIUM)**: Inline Set Decision Logic
   - Implement decision rule for inline vs external sets
   - Add inline set support to manifests
   - **Depends**: P01, P02, P03

5. **P05 (MEDIUM)**: Fast-Path Retrieval Implementation
   - Implement sketch-based overlap computation
   - Add LRU caching for CAS objects
   - **Depends**: P02, P03

6. **P06 (LOW)**: Update Example to Use New Architecture
   - Update FinOps example to use tiered storage
   - Remove CSV re-reading workaround
   - **Depends**: P01-P05

See individual phase files for detailed implementation steps.

## Consequences

### Positive

- **Consistency**: Manifests use same canonicalization as receipts (CBOR)
- **Efficiency**: Chunk sets retrieved from storage, not re-computed
- **Completeness**: Storage API provides all necessary methods
- **Production Ready**: Architecture supports real-world usage
- **Maintainability**: No workarounds or manual serialization

### Negative

- **Breaking Change**: Manifest storage format changes from JSON to CBOR
- **Migration Required**: Existing JSON manifests need migration (if any exist)
- **Complexity**: Additional storage table and methods

### Risks

- **Risk**: CBOR serialization adds complexity
  - **Mitigation**: Reuse existing CBOR utilities from `northroot-receipts`
  - **Mitigation**: Well-tested serialization patterns

- **Risk**: Chunk set storage duplicates manifest data
  - **Mitigation**: Chunk sets are small (just IDs), not full manifest data
  - **Mitigation**: Provides faster access for overlap computation

- **Risk**: Migration of existing manifests
  - **Mitigation**: Support both JSON and CBOR during transition
  - **Mitigation**: Migrate on-demand when manifests are accessed

## Alternatives Considered

### Alternative 1: Keep JSON for Manifests

**Pros**:
- No breaking changes
- Easier debugging (human-readable)

**Cons**:
- Inconsistent with receipts (CBOR)
- Manual serialization workarounds
- Not canonicalized properly

**Decision**: Rejected - consistency and canonicalization are more important.

### Alternative 2: Extract Chunk Sets from Manifests On-Demand

**Pros**:
- No duplicate storage
- Single source of truth (manifest)

**Cons**:
- Slower (requires manifest deserialization)
- More complex overlap computation
- Less efficient for frequent access

**Decision**: Rejected - efficiency and simplicity favor separate chunk set storage.

### Alternative 3: Store Chunk Sets in Manifest Metadata

**Pros**:
- Single storage location
- No separate table needed

**Cons**:
- Still requires manifest deserialization
- Mixes concerns (manifest structure + chunk set)
- Less efficient

**Decision**: Rejected - separate storage provides better performance and separation of concerns.
