# ADR-0010: v0.1.0 Release Readiness Evaluation

Date: 2025-11-12  
Status: Accepted  
Domain: Release

## Context

With ADR-0009 (Hybrid ByteStream/RowMap Evidence Substrate) now complete across all 8 phases, we need to evaluate the overall readiness of the Northroot codebase for a v0.1.0 release. This ADR establishes a comprehensive evaluation framework to assess:

1. **Core Functionality Completeness** - Are all critical features implemented?
2. **Code Quality & Stability** - Is the codebase production-ready?
3. **Documentation & Usability** - Can users effectively use the system?
4. **Testing & Validation** - Is the system sufficiently tested?
5. **Integration & Examples** - Do we have working end-to-end examples?
6. **Release Artifacts** - Are all release materials prepared?

This evaluation will inform the decision to proceed with v0.1.0 release or identify remaining work items.

## Decision Drivers

- **User Value**: v0.1.0 should provide tangible value to early adopters
- **Stability**: Core APIs should be stable enough for integration
- **Completeness**: Critical path features must be functional
- **Quality**: Code quality should meet production standards
- **Documentation**: Users should be able to understand and use the system
- **Maintainability**: Codebase should be maintainable for future development

## Decision

**Decision**: **Option A - Python SDK-First v0.1.0** ✅

**Rationale**:
1. **Developer Ergonomics**: Python SDK provides significantly better developer experience than low-level Rust APIs
2. **Faster Adoption**: Python ecosystem is larger and more accessible than Rust
3. **Launch Readiness**: Python SDK enables meaningful adoption at launch rather than requiring Rust expertise
4. **Property Tests**: Essential for production robustness - catch edge cases that unit tests miss
5. **Real Example**: Demonstrates actual value with real data processing
6. **Container**: Improves developer experience and CI/CD consistency
7. Core functionality is complete and well-tested
8. Establishes release cadence and validation process

## Architecture Decisions

### Python SDK Location

**Decision**: Python SDK will be located in `sdk/northroot-sdk-python/` (not `crates/`)

**Rationale**:
- Clear separation: `crates/` = Rust crates, `sdk/` = language bindings
- Better organization for future SDKs (Rust SDK, JS SDK, etc.)
- Matches architecture gaps analysis recommendation
- Python packaging (`pyproject.toml`, `maturin`) is different from Rust crates

**Structure**:
```
northroot/
├── crates/              # Rust crates only
│   ├── northroot-engine/
│   ├── northroot-receipts/
│   └── ...
├── sdk/                 # Language SDKs
│   └── northroot-sdk-python/  # Python SDK with PyO3
│       ├── Cargo.toml   # For PyO3 build
│       ├── pyproject.toml  # For Python packaging
│       └── src/
└── ...
```

**Note**: The Python SDK can still reference Rust crates via path dependencies:
```toml
# sdk/northroot-sdk-python/Cargo.toml
[dependencies]
northroot-engine = { path = "../../crates/northroot-engine" }
northroot-receipts = { path = "../../crates/northroot-receipts" }
```

### Versioning Strategy

**Decision**: Coordinated versioning for v0.1.0, independent versioning for 0.2.0+

**v0.1.0 Strategy (Coordinated)**:
- All Rust crates: `0.1.0`
- Python SDK: `0.1.0`
- Single release milestone
- Simpler release process
- Clear "v0.1.0 release" for users

**0.2.0+ Strategy (Independent)**:
- Rust crates can version independently (e.g., `0.1.1`, `0.2.0`)
- Python SDK can version independently (e.g., `0.1.1`, `0.2.0`)
- More flexible for post-1.0 maturity
- Better for SDK-specific features

**Implementation**:
- Rust crates: Use `version.workspace = true` for coordinated, or explicit version for independent
- Python SDK: `pyproject.toml` with independent version field
- Release process: Document which packages are included in each release
