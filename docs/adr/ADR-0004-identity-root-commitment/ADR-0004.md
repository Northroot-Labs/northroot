# ADR-003: Identity Root Commitment

**Date:** 2025-11-08  
**Status:** Superseded  
**Context:** We need a deterministic, privacy-preserving way to commit to the set of actors/keys that participated in an execution, independent of trace ordering.

## Context

Execution receipts need to record which identities (actors, keys, DIDs) participated in or are authorized for an execution. This commitment must be:

1. **Deterministic**: Same set of identities → same root
2. **Order-independent**: Reordering identities doesn't change the root
3. **Privacy-preserving**: No PII leakage (trace_id, span commitments excluded)
4. **Reusable**: Same identity set across different traces produces same root
5. **Verifiable**: Supports inclusion proofs for auditing

The identity root is separate from `trace_set_root` and `trace_seq_root`, which commit to span commitments. Identity root commits to the **who**, not the **what**.

## Decision

We implement `identity_root` as a Merkle root over identity records using RFC-6962 style domain separation.

### Identity Record Format

Each identity is represented as a minimal record:

```json
{
  "did": "did:key:z6MkhaXgBZDvotDkL5257faiztiGiC2QtKLGpbnnEGta2doK",
  "kid": "did:key:z6Mk...#key-1",
  "role": "executor",
  "tenant": "urn:tenant:abc"
}
```

- `did`: Decentralized Identifier (required)
- `kid`: Key identifier within the DID (required)
- `role`: Optional role token ("executor", "submitter", "auditor", etc.)
- `tenant`: Optional tenant identifier

### Computation Algorithm

1. **Collect identities**: Gather all identity records observed in the execution
2. **Sort**: Sort by `(did, kid)` lexicographically (bytewise UTF-8)
3. **Leaf hashing**: For each identity record:
   - Canonicalize with JCS (RFC 8785)
   - Leaf hash = `H(0x00 || JCS(identity_record))`
4. **Merkle tree construction**:
   - Pair hashes left-to-right
   - Parent hash = `H(0x01 || left || right)`
   - Odd nodes are promoted upward (standard CT-tree rule)
   - Empty tree root = `H(0x00 || "")`
5. **Root**: Final remaining hash at the top, formatted as `sha256:<64hex>`

### Location in Receipt

- **In execution payload**: `roots.identity_root` (required field)
- **In envelope context**: `ctx.identity_ref` (primary signer/submitter DID, separate from identity set)

### Privacy Considerations

- Identity root does **not** include `trace_id` or `span_commitments`
- This allows the same identity set to be reused across traces
- If a composite binding is needed, derive `exec_root = sha256(JCS({method_shape_root, trace_set_root, identity_root}))` separately

## Consequences

### Pros

- **Deterministic**: Same identities always produce same root
- **Order-independent**: Sorting ensures consistency
- **Privacy-preserving**: No trace-specific data in identity root
- **Reusable**: Same identity set across traces produces same root
- **Verifiable**: Supports Merkle inclusion proofs
- **Standard**: Uses RFC-6962 domain separation (consistent with Merkle Row-Map)

### Cons

- **Additional computation**: Merkle tree construction adds overhead
- **Specification complexity**: Requires clear documentation of leaf format and sorting
- **Migration**: Existing vectors with hardcoded identity_root need regeneration

### Implementation

- `IdentityRecord` struct in `northroot-receipts`
- `identity_root_from_identities()` function in `northroot-receipts`
- Unit tests verify: two identities, remove one, reorder, empty, single, three identities
- Vector generation computes identity_root programmatically

## Alternatives Considered

1. **Simple hash of concatenated DIDs**: Rejected - not order-independent, no structure
2. **Include trace_id in identity root**: Rejected - breaks reusability across traces
3. **Include span_commitments in identity root**: Rejected - mixes "who" with "what", breaks privacy
4. **CBOR canonicalization**: Deferred - JCS is sufficient for now (see ADR-002)

## Migration

1. Update `ExecutionRoots` struct to include `identity_root`
2. Update JSON schema to require `identity_root` in execution receipts
3. Regenerate test vectors with computed identity_root values
4. Update engine tests to compute identity_root from identity records
5. Document identity leaf format in specification
