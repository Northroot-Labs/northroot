# ADR-009: Hybrid ByteStream/RowMap Evidence Substrate with Privacy-Preserving Resolver and Reuse Pipeline
Date: 2025-11-11 
Status: Accepted (Phases 1-6 Complete)

## Context

The v0.1.0 roadmap (see "Core Engine Upgrade to v0.1.0: Hybrid ByteStream/RowMap Architecture") requires the engine to represent data uniformly as byte streams while still exposing row-aware overlays for delta reasoning. Today, the receipts crate only commits to a flat `schema_hash`/`sketch_hash` pair (`crates/northroot-receipts/src/lib.rs:392-406`), and execution receipts make do with a monolithic `data_shape_hash` plus optional chunk manifest pointer (`lib.rs:494-530`). The engine's Merkle Row-Map still uses the ad-hoc `"leaf:"/"node:"` domain separators (`crates/northroot-engine/src/execution/merkle_row_map.rs:58-110`), violating the RFC-6962 alignment the plan calls for. Storage has no notion of encrypted locators or output digests—`crates/northroot-storage/src/sqlite.rs:60-178` shows only raw receipts/manifests with TODOs for PAC/manifest metadata—and therefore cannot satisfy the "resolver keeps secrets, receipts keep proofs" constraint from ADR-001/ADR-002.

Without a formal decision, these gaps risk (a) ambiguous hashing semantics (hard to reason about PAC keys), (b) leaky artifact references that break tenant privacy, (c) duplicate canonicalization paths spread between engine and receipts, contrary to the rules in `docs/ADR_PLAYBOOK.md`, and (d) inefficient reuse checks that require loading full manifests.

## Decision

We adopt the hybrid ByteStream/RowMap substrate described in the plan, scoped across eight phases:

| Phase ID | Description | Status | Documentation |
|----------|-------------|--------|---------------|
| ADR-009-P1 | Engine-internal `DataShape` enum + hash helper | ✅ Complete | - |

| ADR-009-P3 | Refactor Merkle Row-Map and ByteStream manifest builders | ✅ Complete | - |
| ADR-009-P4 | Privacy-Preserving Resolver API | ✅ Complete | [ADR-009-P4](../docs/phases/ADR-009-P4.md) |

### Phase Details

1. **ADR-009-P1: Engine-internal `DataShape` enum + hash helper**  
   - Introduce `crates/northroot-engine/src/shapes.rs` containing `DataShape::{ByteStream, RowMap}` plus supporting enums (`ChunkScheme`, `KeyFormat`, `RowValueRepr`).  
   - Provide `compute_data_shape_hash(&DataShape) -> String` that uses engine-local JCS (per `crates/northroot-engine/src/commitments.rs`) to serialize, while deferring CBOR/JCS exposure at API boundaries to `receipts` adapters (per ADR-002).  
   - Execution code computes PAC inputs from the enum, then injects only the resulting `sha256:<64hex>` into `ExecutionPayload.data_shape_hash` so receipts stay stable.

2. **ADR-009-P2: Extend `ExecutionPayload` to differentiate byte-level commitments vs Merkle overlays**  
   - Add optional fields: `output_digest`, `manifest_root`, `output_mime_type`, `output_size_bytes`, `input_locator_refs`, `output_locator_ref`. All live in `crates/northroot-receipts/src/lib.rs` to keep receipt formats canonical.  
   - `output_digest` is a flat SHA-256 of the materialized bytes (fast cache lookups); `manifest_root` captures RFC-6962 roots over multi-part outputs; `merkle_root` stays reserved for generic integrity proofs.  
   - Locator refs are serialized as the new `EncryptedLocatorRef` struct so that receipts never leak storage URIs, satisfying the privacy boundary laid out in the plan.

3. **ADR-009-P3: Refactor Merkle Row-Map and ByteStream manifest builders behind clear modules**  
   - Move `execution/merkle_row_map.rs` → `rowmap.rs`, swapping the `"leaf:"/"node:"` prefixes for byte prefixes `0x00/0x01` (RFC-6962). Include row normalization rules (UTF-8, LF, trim) and frontier helpers for delta recompute.  
   - Add `cas.rs` for ByteStream manifests: chunking (CDC + fixed), manifest assembly, and RFC-6962 hashing. CAS stays engine-only; receipts just reference manifest hashes.

4. **ADR-009-P4: Introduce resolver contracts for privacy-preserving artifact resolution**  
   - Define `ArtifactResolver` and optional `ManagedCache` traits under `crates/northroot-engine/src/resolver.rs`; engine depends on these traits but never on tenant-specific implementations.  
   - Add manifest root computation helper for multi-part outputs (`crates/northroot-engine/src/delta/manifest_root.rs`).

5. **ADR-009-P5: Summarized manifests for fast overlap computation**  
   - Implement `ManifestSummary` structure with MinHash sketches, HLL cardinality, and optional Bloom filters under `crates/northroot-engine/src/delta/manifest_summary.rs`.  
   - Add compaction tier support (`crates/northroot-storage/src/compaction.rs`) for Hot/Warm/Cold storage optimization.

6. **ADR-009-P6: Storage extensions for encrypted locators, output digests, and manifest summaries**  
   - Extend `ReceiptStore` (`crates/northroot-storage/src/traits.rs`) and the SQLite backend to persist encrypted locator blobs, output digests, manifest summaries, and compaction tiers. All blobs remain opaque to Northroot—only tenants decrypt via the resolver.

7. **ADR-009-P7: Reuse reconciliation flow with summarized manifests**  
   - Implement `ReuseReconciliation` + `check_reuse()` helpers under `crates/northroot-engine/src/delta/reuse.rs` so reuse checks pull summaries (PAC, MinHash, HLL, Bloom) before fetching heavy manifests.  
   - Tie decisions back into `ExecutionPayload.justification` so receipts document overlap `J`, α, and economic deltas.

8. **ADR-009-P8: Helper functions for shape hash computation**  
   - Add data shape hash helpers (`crates/northroot-engine/src/delta/data_shape.rs`) for computing from files, bytes, or multiple inputs.  
   - Add method shape hash helpers (`crates/northroot-engine/src/delta/method_shape.rs`) for computing from code or signatures.

These steps keep canonicalization ownership inside the receipts crate (CBOR, JSON adapters) while allowing the engine to reason over richer shapes without leaking private data, and enable efficient reuse checks via summarized manifests.

## Consequences

**Benefits**
- Deterministic PAC inputs: DataShape hashing now reflects actual ByteStream/RowMap semantics instead of relying on ad hoc schema hashes, reducing cross-run ambiguity.
- Privacy by construction: Receipts hold encrypted locator pointers, and storage never sees tenant URIs in the clear (aligns with "resolver keeps secrets" principle).
- Standards compliance: Using byte prefixes for both RowMap and ByteStream manifests ensures compatibility with external verifiers and simplifies proof sharing.
- Operational clarity: Output digests (flat) vs manifest roots (hierarchical) vs generic `merkle_root` are now explicit, easing cache/reuse tooling.
- Efficient reuse checks: Summarized manifests (MinHash, HLL, Bloom) enable fast overlap estimation without loading full manifests, reducing I/O and latency.
- Storage optimization: Compaction tiers allow cost-effective storage management based on access patterns.

**Trade-offs / Costs**
- Migrating domain-separated hashes invalidates existing RowMap roots; vectors and downstream proofs must be regenerated.
- Additional optional fields increase receipt size slightly, though all are `Option<>` and CBOR already compresses absent fields.
- Introducing resolver traits shifts responsibility to tenants to implement encryption/storage—Northroot must provide reference SDKs to reduce friction.
- CDC chunking and manifest summaries add CPU overhead during ingestion, but this is offset by faster overlap checks later.
- Summarized manifests require additional storage space, but enable faster reuse decisions.

## Alternatives

1. **Keep legacy schema hashes + ad hoc manifests**  
   Rejected: duplicates canonical logic, leaves PAC keys underspecified, and offers no way to prove sub-part reuse or protect artifact URIs.

2. **Push DataShape enum into receipts instead of engine**  
   Rejected after reviewing `docs/ADR_PLAYBOOK.md`: receipts own evidence formats, but execution semantics (chunking, CDC, frontier math) live in `engine`. Keeping the rich enum engine-side while receipts store only hashes preserves dependency direction (receipts ← engine is forbidden).

3. **Use JSON (JCS) everywhere**  
   Rejected because ADR-002 mandates CBOR for receipts; JSON adapters exist only for API boundaries. Engine-local JCS is acceptable for intermediate hashing, but serialized receipts remain CBOR.

4. **Load full manifests for all reuse checks**  
   Rejected: too slow for large datasets. Summarized manifests provide fast overlap estimation, with full manifest loading only when overlap looks promising.

## Architectural Integrity Notes

- **ADR-009-P4 vs ADR-009-P6**: The plan separates resolver API (ADR-009-P4) from storage extensions (ADR-009-P6). The ADR initially combined them, but they are now properly separated to match the plan. Storage extensions (ADR-009-P6) are complete, while resolver traits (ADR-009-P4) are also complete.
- **ManifestSummary**: The struct exists in storage traits (ADR-009-P6), but the engine-side module for generating and using summaries (ADR-009-P5) is not yet implemented. This is intentional—storage provides the persistence layer, engine provides the computation layer.
- **Reuse Pipeline**: ADR-009-P7 depends on ADR-009-P5 (summarized manifests) being complete, as it uses summaries for fast overlap estimation.
- **Helper Functions**: ADR-009-P8 provides convenience functions for PAC key computation, but is not strictly required for core functionality.

This ADR supersedes the implicit assumptions captured in ADR-005 and aligns execution evidence with the privacy and portability goals required for the v0.1.0 release.
