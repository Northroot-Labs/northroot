# ADR-003: Delta Compute Decisions Recorded Under spend.justification

**Date:** 2025-11-08  
**Status:** Superseded  
**Context:** Need to record reuse decisions for audit and transparency

## Context

Delta compute strategies make decisions about whether to reuse previous computation results or recompute. These decisions have economic and technical implications that should be auditable and transparent.

## Decision

Record reuse decisions in the `SpendPayload.justification` field as a `ReuseJustification` struct containing:

- **overlap_j**: Jaccard overlap [0,1] between prior and current chunk sets
- **alpha**: Operator incrementality factor [0,1]
- **c_id**: Identity/integration cost
- **c_comp**: Baseline compute cost
- **decision**: "reuse" | "recompute" | "hybrid"
- **layer**: Semantic level of shape equivalence ("data" | "method" | "reasoning" | "execution")

**Decision rule:** `Reuse iff J > C_id / (α · C_comp)`

**Location:** `northroot-receipts` crate (part of SpendPayload structure)

**Computation:** `northroot-engine` crate (delta compute strategies)

## Consequences

### Pros
- Transparent audit trail of reuse decisions
- Enables economic analysis and optimization
- Supports policy-driven thresholds
- Clear separation: data model (receipts) vs. computation (engine)

### Cons
- Adds optional fields to SpendPayload
- Requires careful versioning if justification structure changes

## Alternatives

**Alternative 1:** Store justification in engine-only structures
- Rejected: Loses auditability, justification should be part of receipt

**Alternative 2:** Separate justification receipt kind
- Rejected: Over-engineered, justification is naturally part of spend

**Alternative 3:** No justification field
- Rejected: Lacks transparency, makes debugging and optimization difficult

## Migration

No migration needed - this is the initial design. Future changes to justification structure should be additive (MINOR version) or breaking (MAJOR version).
