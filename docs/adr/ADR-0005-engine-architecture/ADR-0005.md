# ADR-005: Engine Architecture

**Date:** 2025-01-15  
**Status:** Accepted  
**Context:** Need to document the engine module structure, dependency boundaries, and composition patterns

## Context

The `northroot-engine` crate provides the proof algebra implementation and execution engine for the Northroot system. As the engine has grown to include multiple modules (composition, signature, policy, delta, execution, strategies), we need clear documentation of the architecture, module organization, and design patterns.

## Decision

### Module Organization

The engine is organized into the following modules:

1. **`commitments`**: Core commitment computation utilities
   - SHA-256 hashing with prefix
   - JSON Canonicalization (JCS)
   - Set and sequence root computation

2. **`composition`**: Receipt composition operations
   - Sequential composition validation
   - Parallel (tensor) composition
   - Parent-child link validation
   - Identity receipt generation

3. **`signature`**: Signature verification
   - Ed25519 signature verification
   - Basic DID key resolution (`did:key`)
   - Multi-signature support (future)

4. **`policy`**: Policy validation
   - Policy reference format validation
   - Determinism class enforcement
   - Tool/region constraint stubs (future)

5. **`delta`**: Delta compute operations
   - Overlap computation (Jaccard similarity)
   - Reuse decision logic
   - Chunking utilities

6. **`execution`**: Execution tracking and state
   - Trace ID management
   - Span commitment tracking
   - Execution root computation
   - Merkle Row-Map implementation
   - Execution receipt builder

7. **`strategies`**: Composable compute strategies
   - Strategy trait definition
   - Partition strategy
   - Incremental sum strategy

### Dependency Boundaries

**Engine depends on:**
- `northroot-receipts`: Receipt data model and types
- `northroot-commons`: Shared utilities (future)

**Engine does NOT depend on:**
- `northroot-ops`: Operator manifests (future integration)
- `northroot-policy`: Policy registry (future integration)

### Composition Patterns

1. **Trait-based strategies**: Strategies implement the `Strategy` trait for composability
2. **Builder pattern**: Execution receipts use builder pattern for construction
3. **Error types**: Each module defines its own error types (e.g., `CompositionError`, `SignatureError`, `PolicyError`, `StrategyError`)
4. **Re-exports**: Common items are re-exported at crate root for convenience

### Module Dependencies

```
lib.rs
├── commitments (no dependencies)
├── composition (depends on: commitments, receipts)
├── signature (depends on: receipts)
├── policy (depends on: receipts)
├── delta (depends on: commitments, receipts)
├── execution (depends on: commitments, receipts, delta)
└── strategies (depends on: execution, delta)
```

## Consequences

### Pros

- **Clear separation**: Each module has a single, well-defined responsibility
- **Composability**: Strategies can be combined in pipelines
- **Testability**: Each module can be tested independently
- **Extensibility**: New strategies and modules can be added without breaking existing code
- **No circular dependencies**: Dependency graph is acyclic

### Cons

- **Some duplication**: Error types are defined per module (could be unified in future)
- **Re-export complexity**: Many re-exports at crate root (acceptable for convenience)

## Alternatives

**Alternative 1:** Single monolithic module
- Rejected: Too large, hard to navigate, violates separation of concerns

**Alternative 2:** Separate crates for each major component
- Rejected: Over-engineering for current scope, adds complexity to dependency management

**Alternative 3:** Unified error types in commons
- Deferred: Current per-module errors are fine; can be refactored later if needed

## Implementation Notes

- All modules follow Rust naming conventions
- Public APIs are documented with `///` doc comments
- Tests are co-located with modules (`#[cfg(test)]`) and in `tests/` directory
- Error types implement `std::error::Error` and `std::fmt::Display`
